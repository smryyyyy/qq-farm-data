# 后台闹钟无通知问题排查报告

## 问题现象
PWA 在后台运行或关闭时，闹钟触发时收不到浏览器系统通知。

## 已知修复内容（2026-04-03）
- ✅ sw.js 中已实现 `SHOW_NOTIFICATION` 消息处理
- ✅ app.js 已通过 `registration.active.postMessage()` 发送通知数据
- ✅ Service Worker 每10秒发送 `CHECK_ALARMS` 消息
- ✅ 页面恢复时自动检查过期闹钟
- ✅ visibilitychange 延迟3秒停止语音

## 核心问题分析

### 问题1：触发防重入机制导致通知跳过 ⚠️
**位置**：app.js 第969-970行
```javascript
function triggerAlarm(timer) {
    // 防重入：同一个闹钟只触发一次
    if (triggeredAlarmIds.has(timer.id)) return;
    triggeredAlarmIds.add(timer.id);
```

**问题**：
- 页面在前台时闹钟触发，timer.id 被加入 `triggeredAlarmIds`
- 切换到后台后，Service Worker 发送 `CHECK_ALARMS` 消息
- 页面收到消息调用 `checkAlarmsNow()`，再次检测到同一个过期的 timer
- 由于 timer.id 已经在 `triggeredAlarmIds` 中，直接 return，不再触发通知
- **结果**：后台补触发时通知被跳过

**影响场景**：
1. 前台触发闹钟（收到通知）
2. 但用户可能没看到通知，想要后台补发
3. 由于防重入机制，后台不会再触发

### 问题2：Service Worker 后台检查依赖前台 ⚠️
**位置**：sw.js 第64-79行
```javascript
async function checkAlarmsInBackground() {
  try {
    const clients = await self.clients.matchAll({ type: 'window', includeUncontrolled: true });
    if (clients.length > 0) {
      clients.forEach((client) => {
        client.postMessage({
          type: 'CHECK_ALARMS'
        });
      });
    }
  } catch (error) {
    console.error('[Service Worker] 检查闹钟失败:', error);
  }
}
```

**问题**：
- Service Worker 完全依赖前台页面响应 `CHECK_ALARMS` 消息
- 如果 PWA 被关闭（无活跃 clients），`checkAlarmsInBackground()` 无效
- Service Worker 无法直接访问 `localStorage` 中的闹钟数据
- **结果**：PWA 关闭后完全依赖 GitHub Actions 微信推送

### 问题3：通知发送后无确认机制 ⚠️
**位置**：app.js 第918-926行
```javascript
if (registration?.active) {
    console.log('[通知] 使用 Service Worker 发送通知');
    registration.active.postMessage({
        type: 'SHOW_NOTIFICATION',
        title: '🌾 农场收菜提醒',
        options: options
    });
    console.log('[通知] Service Worker 通知消息已发送');
    return { ok: true, channel: 'service-worker-message' };
}
```

**问题**：
- `postMessage` 是异步的，但没有等待 SW 的确认响应
- 如果 SW 未正确处理消息，前台也不知道
- **结果**：通知可能静默失败

## 解决方案

### 方案A：修复防重入机制（推荐）
**目标**：允许闹钟在页面恢复时重新触发通知

**实现**：
1. 在 `visibilitychange` 时清空 `triggeredAlarmIds`
2. 或者在后台检测时跳过防重入检查

```javascript
document.addEventListener('visibilitychange', () => {
    if (!document.hidden) {
        // 页面恢复时清空防重入集合，允许重新触发通知
        triggeredAlarmIds.clear();
        renderRunningTimers();
        renderAlertsList();
        // ... 现有逻辑
    }
});
```

**优点**：
- 简单直接
- 符合用户预期（页面恢复时补发通知）

**缺点**：
- 可能重复触发已处理过的闹钟（需要检查 timer 是否还在列表中）

### 方案B：Service Worker 直接发送通知（最优）
**目标**：Service Worker 能够独立检查和发送通知，不依赖前台

**实现**：
1. 将闹钟数据存储到 IndexedDB（SW 可以访问）
2. Service Worker 每10秒从 IndexedDB 读取闹钟数据
3. Service Worker 直接检测过期并调用 `showNotification()`

**优点**：
- 完全独立于前台
- PWA 关闭后也能收到通知
- 更可靠的通知机制

**缺点**：
- 需要引入 IndexedDB
- 需要修改数据存储架构
- 开发工作量较大

### 方案C：增强通知确认机制
**目标**：确保通知发送成功

**实现**：
1. Service Worker 收到 `SHOW_NOTIFICATION` 消息后，发送确认响应
2. 前台等待确认，超时则重试或使用页面通知

```javascript
// app.js
const response = await new Promise((resolve) => {
    const channel = new MessageChannel();
    channel.port1.onmessage = (event) => {
        resolve(event.data);
    };
    registration.active.postMessage({
        type: 'SHOW_NOTIFICATION',
        title: '🌾 农场收菜提醒',
        options: options
    }, [channel.port2]);
});
```

```javascript
// sw.js
self.addEventListener('message', (event) => {
  if (event.data && event.data.type === 'SHOW_NOTIFICATION') {
    const { title, options } = event.data;
    self.registration.showNotification(title, options)
        .then(() => {
            // 发送确认响应
            if (event.ports[0]) {
                event.ports[0].postMessage({ ok: true });
            }
        })
        .catch((error) => {
            if (event.ports[0]) {
                event.ports[0].postMessage({ ok: false, error: error.message });
            }
        });
  }
});
```

**优点**：
- 确保通知发送成功
- 增加调试能力

**缺点**：
- 仍依赖前台页面活跃
- PWA 关闭后仍无法发送通知

## 建议实施顺序

1. **立即修复**：方案A（修复防重入机制）
   - 立即见效
   - 解决页面恢复时的补发问题

2. **中期优化**：方案C（增强通知确认）
   - 提升可靠性
   - 便于调试

3. **长期重构**：方案B（IndexedDB + SW 独立通知）
   - 彻底解决 PWA 关闭后的通知问题
   - 需要较大架构调整

## 其他潜在问题

### 问题4：Service Worker 激活时机
**可能问题**：Service Worker 还未完全激活就尝试发送通知

**检查**：
- 控制台日志中 `registration?.active?.state` 是否为 `activated`
- 是否有 SW 激活失败的日志

### 问题5：通知权限问题
**可能问题**：通知权限在后台被撤销或过期

**检查**：
- Notification.permission 在触发时的值
- 是否有权限被撤销的提示

## 总结

**核心问题**：防重入机制导致后台补触发时通知被跳过

**最快解决方案**：在 `visibilitychange` 时清空 `triggeredAlarmIds`

**最优长期方案**：引入 IndexedDB + Service Worker 独立通知机制

**当前建议**：先实施方案A，立即见效；再逐步实施方案C和B
